## 1️⃣ ENVIRONMENT VARIABLES STRUCTURE

The execution environment is strictly segmented between frontend client boundaries and secure backend server (Cloud Functions) execution boundaries.

### Backend Execution Variables (Firebase Secret Manager / Cloud Functions)

`STRIPE_SECRET_KEY=`
*   **Definition:** Live/Test secret key from Stripe Developer Dashboard.
*   **Location:** Google Cloud Secret Manager, injected into Firebase Cloud Functions at runtime.
*   **Usage:** Used exclusively by `onUserCreate`, `createCheckoutSession`, and `handleStripeWebhook` to authenticate API calls to Stripe. Never exposed to the client.

`STRIPE_WEBHOOK_SECRET=`
*   **Definition:** Unique signing secret generated by Stripe for the registered webhook endpoint.
*   **Location:** Google Cloud Secret Manager.
*   **Usage:** Used by `handleStripeWebhook` to cryptographically verify that incoming POST requests originated directly from Stripe.

`FIREBASE_SERVICE_ACCOUNT=`
*   **Definition:** JSON key file granting administrative bypass privileges to the Firebase ecosystem.
*   **Location:** Encrypted environment variable in CI/CD pipeline.
*   **Usage:** Used for programmatic deployment of Cloud Functions and offline administrative scripts.

### Frontend Execution Variables (Antigravity Hosting / Next.js .env.local)

`STRIPE_PRICE_STANDARD=`
*   **Definition:** The specific `price_xyz123` ID defining the 7-day trial and recurring subscription parameters.
*   **Location:** Frontend build environment (`NEXT_PUBLIC_STRIPE_PRICE_STANDARD`).
*   **Usage:** Passed as a payload parameter from the frontend client to the backend `createCheckoutSession` callable function.

`FIREBASE_PROJECT_ID=`
*   **Definition:** The unique alphanumeric identifier for the Firebase instance (e.g., `savehxpe-prod`).
*   **Location:** Frontend build environment (`NEXT_PUBLIC_FIREBASE_PROJECT_ID`).
*   **Usage:** Used by the Firebase Client SDK to initialize the connection to Auth and Firestore.

`ANTIGRAVITY_HOSTING_URL=`
*   **Definition:** The apex domain where the application is deployed (e.g., `https://app.savehxpe.com`).
*   **Location:** Both frontend and backend configuration.
*   **Usage:** Used by the backend to generate dynamic `success_url` and `cancel_url` redirects for Stripe Checkout.

## 3️⃣ STANDARD TRIAL FLOW

**Step 1: User upgrades**

Trigger: User clicks "Upgrade to Standard" button on `/dashboard`.

Function: Client invokes Firebase Callable Function `createCheckoutSession({ priceId: STRIPE_PRICE_STANDARD })`.

Database Write: None.

Stripe Interaction: Backend requests `stripe.checkout.sessions.create()`, passing `customer: stripeCustomerId` and trial parameters. Stripe returns `session.url`.

UI Change: Client receives `session.url` and redirects window location to Stripe Checkout.

**Step 2: Trial activates**

Trigger: User enters payment details on Stripe Checkout and submits.

Function: Stripe verifies card and activates subscription in `trialing` status.

Database Write: None (yet).

UI Change: Stripe redirects user to `success_url` (`/dashboard?upgrade=success`).

Stripe Interaction: Stripe backend fires POST request to `handleStripeWebhook` endpoint.

**Step 3: Webhook processing & Vault unlock**

Trigger: Webhook endpoint receives `customer.subscription.created`.

Function: `handleStripeWebhook` verifies signature, extracts `customer_id`, queries Firestore for user.

Database Write: `userRef.update({ 'tier.current': 'STANDARD', 'trialEndsAt': timestamp, 'xp.multiplier': 1.5 })`.

Stripe Interaction: None.

UI Change: Real-time Firestore listener on client detects `tier.current === 'STANDARD'`. Vault locked overlay is removed. Trial countdown component mounts and begins rendering time remaining. XP multiplier UI indicator activates.

## 4️⃣ WEBHOOK CONTROL CENTER

All financial state mutations originate from Stripe and are enforced via the `handleStripeWebhook` Cloud Function.

```
// EXACT LOGIC TREE
const event = stripe.webhooks.constructEvent(req.rawBody, sig, STRIPE_WEBHOOK_SECRET);
const subscription = event.data.object;
const userRef = db.collection('users').where('stripeCustomerId', '==', subscription.customer);

switch (event.type) {
  
  case 'customer.subscription.created':
    // Trigger: Checkout success.
    // Action: Set tier, define trial, increase XP capacity.
    update payload = {
      'tier.current': 'STANDARD',
      'subscriptionStatus.id': subscription.id,
      'subscriptionStatus.status': subscription.status, // 'trialing'
      'trialEndsAt': subscription.trial_end,
      'xp.multiplier': 1.5
    };
    db.write(payload);
    break;

  case 'customer.subscription.updated':
    // Trigger: Trial converts to active, or user updates payment method.
    // Action: Maintain tier, update period end, nullify trial if converted.
    update payload = {
      'subscriptionStatus.status': subscription.status,
      'subscriptionStatus.currentPeriodEnd': subscription.current_period_end,
      'trialEndsAt': subscription.status === 'active' ? null : subscription.trial_end
    };
    db.write(payload);
    break;

  case 'customer.subscription.deleted':
    // Trigger: User manually cancels and period ends, or admin cancels.
    // Action: Revert tier, disable multiplier, lock vault.
    update payload = {
      'tier.previous': 'STANDARD',
      'tier.current': 'FREE',
      'subscriptionStatus.status': 'canceled',
      'xp.multiplier': 1.0,
      'trialEndsAt': null
    };
    db.write(payload);
    break;

  case 'invoice.payment_failed':
    // Trigger: Automatic charge fails post-trial or on renewal.
    // Action: Revert tier, restrict vault.
    update payload = {
      'tier.previous': 'STANDARD',
      'tier.current': 'FREE',
      'subscriptionStatus.status': 'past_due',
      'xp.multiplier': 1.0
    };
    db.write(payload);
    break;

  default:
    return status(200);
}
```

## 5️⃣ FIRESTORE RULE ENFORCEMENT FLOW

Security rules are enforced at the Firestore routing layer. Requests violating these rules are rejected before execution, returning a `permission_denied` error to the client.

### Tier Protection Logic

**Objective:** Prevent frontend clients from modifying their own access level.

**Rule Enforcement:**

**Flow:** Frontend attempts `db.doc('users/uid').update({ tier: { current: 'PREMIUM' } })`.

**Result:** Firestore security rules intercept. Key `tier` is in the `affectedKeys` diff. Write is REJECTED.

### XP Increment Logic

**Objective:** Prevent client-side XP manipulation or spoofing.

**Rule Enforcement:**

**Flow:** User completes an action (e.g., listens to track). Frontend calls `grantActionXP()` Cloud Function.

**Execution:** Cloud Function verifies action validity, calculates `base_xp * userDoc.xp.multiplier`, and executes `admin.firestore().update()`.

**Result:** Because the Cloud Function operates via `firebase-admin`, it bypasses client security rules. XP is incremented safely.

## 6️⃣ UI RENDER LOGIC

The React/Next.js application state relies on the live stream of the user document.

### `if user.tier === "FREE"`

**Dashboard:** Renders `UpgradePromptHeader`. Trial countdown is hidden.

**Vault:** Renders `VaultLockedOverlay` component. CSS `backdrop-filter: blur` applied to background content. Content array is sliced to only show `item.public === true`.

**Dashboard:** Renders `TrialCountdownBanner` (if `trialEndsAt` > `now`). Renders active subscription management link.

**Vault:** Overlay is destroyed. Full `collectibles` array mapped and rendered. Audio stems become interactive.

**XP Display:** Renders premium purple XP bar. Badge displaying `1.5x` is rendered next to the total score.

### `if user.tier === "PREMIUM"`

**Dashboard:** Renders custom black/gold theme. Mounts `ConciergeChatWidget` for direct contact.

**Vault:** Overlay is destroyed. Renders `PreReleaseSection` component bypassing standard date locks.

**XP Display:** Renders diamond XP bar. Badge displaying `3.0x` is rendered.

## 7️⃣ ANTIGRAVITY DEPLOYMENT FLOW

Execution of updates to the production environment follows a strict sequence.

**Environment Variable Connection:**

Developer pushes code to `main` branch.

CI/CD pipeline initiates.

Pipeline pulls `NEXT_PUBLIC_*` variables from repository secrets and injects them into the Next.js static build process.

Pipeline utilizes `FIREBASE_SERVICE_ACCOUNT` to authenticate Firebase CLI.

**Firebase Functions Deployment:**

Pipeline executes `firebase deploy --only functions,firestore:rules`.

Google Cloud provisions new Node.js containers.

Containers retrieve `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET` directly from Google Secret Manager at cold start.

**Webhook Endpoint URL Configuration:**

Upon successful function deployment, Firebase returns the live function URL (e.g., `https://us-central1-savehxpe.cloudfunctions.net/handleStripeWebhook`).

This URL is manually or programmatically registered in the Stripe Dashboard under Developers > Webhooks.

**Frontend Deployment to Antigravity:**

Next.js build completes.

Static files and edge functions are pushed to Google Antigravity edge nodes.

Frontend initialization script loads `FIREBASE_PROJECT_ID` and establishes a WebSocket connection to Firestore.

## 8️⃣ FAILURE SCENARIOS

System fallback logic mapped strictly to triggers and database mutations.

### Scenario A: Card Declines on Upgrade

**Trigger:** User enters invalid card in Stripe Checkout.

**Stripe Interaction:** Stripe validates card locally. Rejects charge/authorization.

**UI Change:** Stripe Checkout UI displays red error text ("Card declined"). User remains on Stripe page until resolved or canceled.

**Database Write:** None. User remains `FREE`.

### Scenario B: User Cancels Subscription Mid-Trial

**Trigger:** User clicks "Cancel" in Stripe Customer Portal.

**Stripe Interaction:** Stripe updates subscription `cancel_at_period_end = true`.

**Webhook:**`customer.subscription.updated` fires.

**Function:**`handleStripeWebhook` catches event.

**Database Write:**`userRef.update({ 'subscriptionStatus.cancelAtPeriodEnd': true })`.

**UI Change:** Dashboard renders "Trial will expire on \[Date\]". Vault remains UNLOCKED until absolute expiration.

### Scenario C: Trial Expires

**Trigger:** Trial period ends. Stripe attempts to charge card.

**Success Logic:** Charge succeeds. `customer.subscription.updated` fires (`status: active`). `trialEndsAt` set to `null`. User remains `STANDARD`.

**Failure Logic (Card now invalid):** Charge fails. `invoice.payment_failed` fires.

**Function:**`handleStripeWebhook` processes failure.

**Database Write:** Update payload changes `tier.current` to `FREE`, sets `xp.multiplier` to `1.0`.

**UI Change:** Real-time listener immediately locks Vault and renders `VaultLockedOverlay`.

### Scenario D: Webhook Fails (Network Timeout)

**Trigger:** Stripe fires webhook. Firebase Cloud Functions is undergoing cold start or outage, returning 500/timeout.

**Stripe Interaction:** Stripe registers failure. Initiates exponential backoff retry protocol (retries up to 3 days).

**Fallback Logic:**

If a user trial expires logically (`trialEndsAt` < `now`) but the webhook hasn't successfully processed the downgrade, the Frontend `TrialCountdownBanner` logic will fall back to `FREE` state render rules locally if `now` exceeds the timestamp by > 2 hours grace period.

Once the webhook successfully retries, the database syncs, making the state persistent across all devices.
